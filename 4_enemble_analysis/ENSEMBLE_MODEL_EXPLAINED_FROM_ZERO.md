# Ensemble Model Explained From Zero

This document explains your ensemble analysis in very simple language.

It is written for someone who knows nothing about ML, sentiment analysis, or ensemble learning.

## 1) What is an ensemble model?

An ensemble model means:
- Do not trust only one model.
- Combine multiple model outputs.
- Final prediction comes from group decision.

Simple real-life analogy:
- If 1 doctor gives diagnosis, you may still be unsure.
- If 3 doctors give opinion and 2 agree, confidence is higher.
- Ensemble works the same way for models.

In your project, you combine sentiment outputs from:
- VADER
- BERT
- Word Cloud proxy

## 2) Why ensemble is needed in your project

Your project analyzes product reviews and maps sentiment to ratings.

Each single model has strengths and weaknesses:
- BERT: strong accuracy and trend capture, but can miss class balance.
- VADER: simple and fast, but less context-aware than deep models.
- Word Cloud proxy: explainable and keyword-driven, but simplistic.

Why ensemble helps:
- Reduces dependence on one model failure.
- Improves stability of predictions.
- Gives a more robust final sentiment signal for dashboards.
- Better for practical decision making (product pain points and strengths).

## 8.1 Hard Voting

Definition:
- For each review, collect labels from models.
- Final label = class with majority votes.

In your code:
- Uses labels from VADER and Word Cloud.
- Includes BERT if available.
- Tie-break rule: prefer VADER label.

Formula idea:
- y_hat = mode([label_vader, label_bert, label_wc])

Then converts label to numeric rating midpoint:
- Positive -> 4.5
- Neutral -> 3.0
- Negative -> 1.5

Easy example:
- VADER = Positive
- BERT = Negative
- Word Cloud = Positive
- Majority = Positive
- Hard-vote predicted rating = 4.5

Why useful:
- Very interpretable.
- Works even if one model is noisy.
- Good when you need class decision quickly.

## 8.2 Soft Voting (weighted average)

Definition:
- Instead of voting on labels, combine continuous predicted ratings.

In your code, if BERT is available:
- soft_pred = 0.40*VADER + 0.40*BERT + 0.20*WordCloud

If BERT not available:
- soft_pred = 0.60*VADER + 0.40*WordCloud

Then convert rating to class:
- rating >= 3.75 -> Positive
- rating <= 2.25 -> Negative
- else -> Neutral

Easy example:
- VADER rating = 4.2
- BERT rating = 3.8
- WC rating = 3.4
- soft_pred = 0.40*4.2 + 0.40*3.8 + 0.20*3.4
- soft_pred = 1.68 + 1.52 + 0.68 = 3.88
- Final label = Positive

Why useful:
- Uses intensity information, not only class.
- Usually smoother than hard voting.
- Better when models disagree slightly but same general direction.

## 8.3 Bagging (Bootstrap Aggregating)

Definition:
- Create many random samples from data (with replacement).
- Predict on each sample.
- Aggregate predictions (mean).

In your code:
- n_bootstraps = 50
- sample fraction = 0.80
- For each bootstrap sample, compute weighted average rating.
- Final bagging rating = mean across 50 bootstrap predictions.
- Also compute prediction standard deviation (uncertainty).

What this means for your 4,000-row CSV:
- Each bootstrap sample uses about 80% of the rows, so around 3,200 rows at a time.
- Sampling is done with replacement, so the same review can appear more than once in one sample.
- Because of that, some reviews are repeated, and some reviews are not picked in that round.
- This process is repeated 50 times, so the script creates 50 slightly different views of the same 4,000 reviews.
- For each view, the code recalculates the weighted sentiment score.
- Then it averages those 50 scores to get one final bagging rating for each review.
- It also calculates the standard deviation across those 50 scores.

Simple interpretation:
- If the 50 scores are close to each other, the prediction is stable.
- If the 50 scores spread out a lot, the prediction is uncertain.
- So bagging is not changing the CSV data itself; it is repeatedly resampling the rows to test how stable the sentiment result is.

Important meaning:
- Lower std dev means prediction is stable across random resampling.
- Higher std dev means model confidence is weaker for those reviews.

Easy example (conceptual):
- Review A predicted ratings over 5 bootstraps: [4.0, 4.1, 3.9, 4.0, 4.0]
- Mean = 4.0
- Std dev is low -> stable prediction.

Why useful:
- Tests robustness.
- Reduces impact of sample noise.
- Gives reliability information, not just final score.

## 9) Cumulative rating analysis (device level)

In your project, "cumulative" means we combine all review-level results for one device and summarize them as one final device-level picture.

So instead of looking at one review at a time, the code groups all reviews of the same device and calculates:
- Number of reviews for that device
- Actual mean rating
- Actual mode rating
- Actual percent positive reviews
- VADER mean rating and percent positive
- BERT mean rating and percent positive
- Word Cloud mean rating and percent positive

What this means in simple language:
- If a device has many positive reviews, its cumulative score will also look positive.
- If the predicted mean rating is close to the actual mean rating, that method is behaving well.
- If the predicted value is far from the actual value, that method is overestimating or underestimating the device quality.

Why this matters in your project:
- Your final goal is not only to label single reviews.
- You want a product-level summary for iPhone 15, iPhone 16, and iQOO Z10.
- This cumulative table helps compare how each method sees the overall device sentiment.

Easy example:
- Device X has actual mean rating = 4.3
- VADER mean rating = 4.1
- BERT mean rating = 4.2
- Word Cloud mean rating = 3.9
- Since these values are close, the device-level summary is stable and reliable.

So this section is basically the final summary of all review predictions for each product.

## 7) Metrics used and why they matter

Your ensemble summary compares methods using:

1. Accuracy
- Correct label percent.
- Useful for high-level classification success.

2. Macro F1
- Average F1 across Positive, Neutral, Negative.
- Useful for class balance fairness.

3. RMSE
- Numeric rating error magnitude.
- Lower is better.

4. Pearson r
- Linear correlation between predicted and actual rating.
- Higher positive is better.

5. Spearman rho
- Rank-order correlation.
- Useful for sorting reviews by severity/satisfaction.

6. Bagging consistency (mean std)
- Measures stability of bagging predictions.
- Lower is better (more robust).

## 9) One full toy example from start to finish

Review text:
- "Display is amazing but battery drains quickly"

Suppose model outputs:
- VADER label Positive, rating 3.9

Hard voting:
- Labels are all different (Positive, Neutral, Negative)
- Tie-break in your code prefers VADER

Soft voting:

- soft = 1.56 + 1.36 + 0.56 = 3.48
- Soft label = Neutral (between 2.25 and 3.75)

Bagging (concept):
- Repeat weighted sampling and averaging 50 times.
- Suppose mean bagging rating = 3.52 and std = 0.12
- Label = Neutral
- Low std suggests stable prediction around Neutral.

If true user rating is 3:
- Hard may over-predict positivity.
- Soft and bagging may be closer.

This is exactly why comparing methods is necessary.

## 10) Limitations to remember

1. If a base model is missing, that branch contributes less or is skipped.
2. Threshold choices (3.75 and 2.25) affect label conversion.
3. Hard-vote tie-break toward VADER can bias some cases.
4. Bagging here evaluates stability mainly on deterministic model outputs.
5. Ground truth comes from rating, which may not always match review text.
